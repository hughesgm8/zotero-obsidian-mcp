## [~16:30] Customisable Summary Sections (closes GitHub issue #1)

- **File Modified**: `src/types.ts`, `src/settings.ts`, `src/paper-importer.ts`, `styles.css`
- **Change**: Summary sections for Smart Import are now fully user-configurable. A new structured list in Settings (Smart Import → Summary sections) lets users add, delete, reorder (↑/↓), and edit the name and instructions for each section. The prompt builder and note builder are now fully dynamic — no hardcoded section names anywhere.
- **Logic Update**:
  ```
  types.ts:
    new SummarySection { name, instructions }
    DEFAULT_SUMMARY_SECTIONS = [Summary, Interesting Takeaways, Questions, Relevance]
    ZoteroMCPSettings.summarySections: SummarySection[]

  paper-importer.ts:
    buildImportPrompt():
      iterate summarySections → dynamic section list in prompt
    buildNote() + generateSummaryMarkdown():
      iterate summarySections → render sections[s.name] for each
    parseLLMSections():
      heading regex: /^#{1,6}\s+(.+)$/  ← was /^##\s+(.+)$/

  settings.ts:
    renderSections() closure → re-renders card list on add/delete/reorder
    delete uses filter() not splice() to avoid mutating DEFAULT_SUMMARY_SECTIONS
    reset clones DEFAULT_SUMMARY_SECTIONS with .map(s => ({...s}))
  ```
- **Reason**: Users like methods-focused researchers may want custom sections. The old hardcoded approach also had a fragile conditional for the Relevance section (only shown when researchDescription was set) — the new dynamic system removes all special-casing.

## [~15:00] Always-On Hybrid Search — Per-Token Keyword Search with Three-Tier Ranking

- **File Modified**: `src/orchestrator.ts`
- **Change**: Replaced the old single semantic search with a parallel hybrid search that runs semantic search and per-token keyword search simultaneously. Results are merged using a principled three-tier ranking system. Removed the previous LLM-based keyword extraction (`extractSearchKeywords`) and the broken `zotero_advanced_search` approach.
- **Logic Update**:
  ```
  query():
    Run in parallel:
      - zotero_semantic_search(question)
      - keywordSearch(question)  ← new

    Three-tier merge:
      tier1 = semanticKeys ∩ keywordKeys   # matched both — highest confidence
      tier2 = keywordKeys - semanticKeys   # keyword-only — specific identifier match
      tier3 = semanticKeys - keywordKeys   # semantic-only — conceptually related
      itemKeys = [...tier1, ...tier2, ...tier3].slice(0, 25)

  keywordSearch(question):
    tokens = question.split(/[\s\W]+/).filter(len >= 3)
    for each token (in parallel):
      zotero_search_items(query=token, qmode="titleCreatorYear", limit=5)
    merge + dedupe all returned item keys
  ```
- **Reason**: Semantic search fails to surface papers when the query contains specific identifiers (filenames, author names, acronyms) the embedding model can't represent. Example: "How effective are AGENTS.md files?" never returned the Gloaguen et al. AGENTS.md paper from semantic search alone.

  **Why `zotero_advanced_search` was abandoned**: That tool creates a temporary saved search via `POST /api/users/0/searches` — but the local Zotero API (port 23119) does not support that endpoint. It only works with the Zotero Web API (cloud), which requires an API key. Our setup (and Claude Desktop's) uses `ZOTERO_LOCAL=true`.

  **Why Claude Desktop handles these queries**: Claude the LLM, given direct MCP tool access, naturally decomposes queries into targeted single-keyword searches. Our deterministic pipeline had to replicate this behaviour explicitly.

  **Why per-token works**: A single-word query to `zotero_search_items` cannot fail AND-logic. `q=AGENTS` finds any paper with "AGENTS" in the title, creator, or year — confirmed working via direct API test (`curl http://localhost:23119/api/users/0/items?q=AGENTS` returned PNIRHMTG immediately).

  **Why the old slice(0,15) broke it**: Semantic search returns ~10 results. These filled the front of the merged array. The AGENTS paper, being keyword-only (tier3 under the old scheme), ended up at position ~17 and was cut off. The new three-tier ranking ensures keyword-specific matches (tier2) appear before semantic-only results (tier3), so a strong keyword match is never buried behind 10 unrelated semantic results.
